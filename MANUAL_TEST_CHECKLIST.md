# 全新 Windows 电脑端到端功能测试清单

本文是一份黑盒功能验收流程。测试人员假设自己是一名普通用户，使用一台从未安装过 WSL、Nekro Agent 和本启动器的新电脑，从安装启动器开始，完整验证首次部署、日常使用、更新、备份恢复和卸载。

本清单只判断用户能看到和操作到的结果，不要求阅读代码、检查内部函数、执行代码质量工具或修改配置文件制造故障。

## 1. 测试结果与记录

结果定义：

- `PASS`：操作成功，页面、提示和最终功能都符合预期。
- `FAIL`：功能不可用、界面假死、提示与实际结果不一致、数据丢失或状态串用。
- `BLOCKED`：因测试机、网络或外部服务原因无法继续，必须写明原因。
- `N/A`：本轮版本不提供该功能，并经负责人确认。

## 2. 测试机准备

### 2.1 主线测试机

- [ ] 使用 x64 Windows 11 新系统；发布前再在 Windows 10 22H2 执行至少一轮主线。
- [ ] BIOS/UEFI 虚拟化已开启，Windows 账号可以在需要时批准管理员权限。
- [ ] 电脑未安装 WSL、Docker Desktop、Nekro Agent 启动器，也没有 `NekroAgent` WSL 发行版。
- [ ] `%LOCALAPPDATA%\NekroAgent` 不存在，没有上一轮启动器配置和浏览器登录状态。
- [ ] 系统盘和运行环境目标盘空间充足；至少满足向导提示的 10 GB 可用空间要求。
- [ ] 首次主线使用稳定网络，默认端口 `8021` 和 `6099` 未被其他程序占用。
- [ ] 已准备本次正式发布的 `NekroAgent-Setup.exe`，不使用源码启动代替安装包。

可在 Windows“已安装的应用”“启用或关闭 Windows 功能”和终端中的 `wsl -l -v` 辅助确认测试机确实没有旧环境。这属于测试前环境确认，不作为代码检查。

### 2.2 推荐快照

异常分支应从对应快照单独执行，避免前一个失败污染后续结果。

| 快照 | 状态 |
| --- | --- |
| S0 | 纯净 Windows，未安装 WSL 和启动器 |
| S1 | WSL 功能已启用，Windows 尚未重启 |
| S2 | WSL 可用，但未创建 `NekroAgent` 运行环境 |
| S3 | 运行环境已导入，但 Docker 安装尚未完成 |
| S4 | WSL、Docker、Compose 已就绪，尚未部署实例 |
| S5 | 默认 Lite stable 实例部署完成 |
| S6 | Lite 与 NapCat 两个实例部署完成 |

## 4. 第二阶段：从零安装 WSL

- [ ] **F06 首次环境检测** — 在“检测环境”页等待检测结束。预期 WSL 显示未安装或未完成启用；运行环境、Docker、Compose 不应假报就绪；主按钮显示安装 WSL 的操作。

## 5. 第三阶段：创建 NekroAgent 运行环境

- [ ] **F13 进入创建页** — 点击“创建运行环境”。预期进入独立创建步骤，显示安装位置、空间要求和操作说明。

- [ ] **F15 开始创建** — 点击开始创建。预期安装位置和返回操作被合理锁定，避免创建中途重复提交；界面显示连续进度而不是无响应。

- [ ] **F17 导入和系统配置** — 下载完成后继续观察发行版导入、systemd/WSL 配置。预期步骤顺序清楚，不要求用户手工打开 WSL 终端执行命令。
- [ ] **F18 Docker 安装** — 等待 Docker 和 Docker Compose 安装。预期安装由向导自动完成；日志或状态能说明正在处理的动作，完成前不会提前进入部署成功页。
- [ ] **F19 创建完成** — 运行环境创建完成后继续。预期 WSL、运行环境、Docker、Compose 均显示就绪，并自动进入版本选择或允许点击下一步。
- [ ] **F20 退出后恢复** — 在部署前关闭启动器再重新打开。预期已创建的运行环境被正确识别，不重新下载或重复导入。

## 6. 第四阶段：首次 Lite stable 部署

- [ ] **F21 选择版本** — 首轮选择稳定版和 Lite/精简模式。预期选项说明清楚，不默认开启 preview 等高级频道。
- [ ] **F22 默认实例配置** — 实例名留空，使用默认实例；先保留默认 Nekro 端口 `8021`。预期页面展示的数据路径和访问地址与默认实例一致。
- [ ] **F23 配置确认** — 点击开始部署。预期在端口、路径和实例信息有效时进入镜像测速；同一个点击不会创建两个部署任务。
- [ ] **F24 镜像源测速** — 等待测速完成，展开详情并执行一次重新测速。预期逐个显示可用性或延迟，健康网络下自动选择可用源；详情和按钮在测速后恢复。

- [ ] **F26 可选组件** — 首轮按最短核心路径跳过可选的 Claude Code 沙盒组件。预期跳过不会影响 Nekro Agent 核心服务部署。
- [ ] **F27 镜像拉取** — 观察镜像名称和进度。预期长时间操作有持续反馈，窗口不卡死；拉取结束后自动继续，而不是要求用户重复点击。
- [ ] **F28 服务启动** — 等待 Compose 和健康检查完成。预期状态从部署中变为服务就绪；不能在 WebUI 实际不可访问时提前报告完成。
- [ ] **F29 凭据窗口** — 记录访问地址、`admin`、管理员密码、OneBot token。预期内容非空且可复制；复制结果与界面一致。服务尚未就绪时关闭操作应受控，成功或明确失败后才能结束等待。

## 8. 第六阶段：停止、退出、托盘与重启恢复

- [ ] **F39 取消停止** — 点击关闭服务，在确认框先取消。预期服务继续运行，WebUI 仍可访问。
- [ ] **F40 停止服务** — 再次操作并确认。预期状态最终变为已停止，WebUI 不可访问；运行环境、凭据和数据 A 不被删除。
- [ ] **F41 重新启动服务** — 点击一键部署/启动。预期不重新安装 WSL 或 Docker，服务恢复运行，原凭据和数据 A 仍可使用。
- [ ] **F42 最小化到托盘** — 服务运行时点击窗口关闭并选择最小化到托盘。预期窗口隐藏但 WebUI 继续可访问；双击托盘图标和“显示主窗口”都能恢复。
- [ ] **F43 托盘退出取消** — 从托盘选择退出，在确认框取消。预期启动器和服务保持不变。
- [ ] **F44 停止服务并退出** — 再次退出并选择停止服务。预期等待停止完成后启动器进程退出，WebUI 不可访问；停止失败时程序保持打开并给出提示。
- [ ] **F45 冷启动恢复** — 从快捷方式重新启动。预期不进入首次向导，自动识别已有实例并恢复服务；总览显示真实状态，数据 A 和凭据不变。
- [ ] **F46 启动器异常退出** — 服务运行时从任务管理器结束 `NekroAgent.exe`，确认 WebUI 仍可访问，再重新启动。预期启动器接回已有服务，不重复部署、不重置凭据，也不因 daemon 端口占用崩溃。
- [ ] **F47 WSL 关闭后恢复** — 正常退出启动器，在 Windows 终端执行 `wsl --shutdown`，再启动。预期运行环境、Docker 和服务被重新拉起；失败时提示具体动作并允许重试，不能假报运行中。
- [ ] **F48 Windows 重启** — 保持现有实例后重启 Windows，再手工启动启动器。预期 WSL、Docker、实例和数据 A 恢复，界面不长期停在启动中。

## 13. 第十一阶段：实例删除、启动器卸载与重新安装

“卸载清理”、卸载 Windows 启动器和彻底删除 WSL 运行环境是三件不同的事，必须分别验证。

### 13.1 删除实例

- [ ] **R01 删除非当前实例** — 当前为 A 时选择卸载清理并选 B，先取消一次再确认。预期取消后 B 仍可用；确认后 B 服务和部署目录被移除，B 从列表消失，A 完全不受影响。
- [ ] **R02 数据目录保留** — 删除前在 B 的数据目录放入 `manual-test-marker.txt`。预期删除实例后该数据目录和 marker 按确认框承诺保留。不要把 Docker 数据库 volume 保留写成预期。
- [ ] **R03 同名重新部署** — 使用原 B 名称和端口重新部署。预期可以重新创建实例，保留的数据目录 marker 仍在；旧数据库对象不作为必须保留项。
- [ ] **R04 删除当前实例** — A 为当前、B 存在时删除 A。预期自动切到 B，B 的页面、日志、浏览器、凭据和数据正常，不显示 A 的残留状态。
- [ ] **R05 删除最后实例** — 删除最后一个实例。预期回到未部署/首次运行状态；再次点击卸载清理应提示没有实例。`wsl -l -v` 仍应看到 `NekroAgent` 发行版，证明删除实例不等于删除运行环境。
- [ ] **R06 Docker 不可用时删除** — 停止 Docker 后尝试删除一次性实例。预期失败时显示可诊断提示，不误报移除完成；恢复 Docker 后可以重试。

### 13.2 只卸载 Windows 启动器

- [ ] **R07 卸载启动器** — 在 A/B 正常运行的快照中，从 Windows“设置 → 已安装的应用”卸载启动器。预期启动器进程、安装目录、快捷方式、开机启动项、启动器设置和内置浏览器资料被删除。
- [ ] **R08 WSL 数据保留** — 卸载启动器后运行 `wsl -l -v`。预期 `NekroAgent` 发行版仍存在，WSL 内部署和业务数据未被启动器卸载器删除。浏览器登录态和启动器偏好不要求保留。
- [ ] **R09 重新安装接管** — 重新安装并启动。预期首次扫描发现已有部署并提供迁移/接管入口；完成接管后原 WebUI 业务数据可访问，不要求重新创建 WSL。

### 13.3 彻底清除运行环境

- [ ] **R10 检查产品入口** — 删除全部实例后检查界面是否提供“卸载整个 WSL 运行环境”。当前版本没有完整入口时，应记录为功能缺口，不能把“卸载清理”当成彻底卸载通过。
- [ ] **R11 人工清除测试环境** — 备份需要保留的数据后，在 Windows 终端明确执行 `wsl --unregister NekroAgent`。预期发行版和其中未备份数据被删除，本地服务不可访问。
- [ ] **R12 环境消失后的启动器** — 清除发行版后重新启动启动器。预期显示运行环境未创建并允许重新进入创建流程，不崩溃、不假报服务运行中。
- [ ] **R13 最终卸载** — 再从 Windows 设置卸载启动器。预期任务管理器、快捷方式、开始菜单、已安装应用和启动应用中均无启动器；外部导出的备份不被删除。
