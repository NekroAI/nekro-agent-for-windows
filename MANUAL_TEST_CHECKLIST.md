# 人工测试 Checklist

## 使用说明

- `实际结果` 填写通过、不通过，或记录现象。
- 建议优先执行混合频道、多实例切换、失败回滚这几组用例。

## 测试清单

| ID | 场景 | 操作 | 预期结果 | 实际结果 |
|---|---|---|---|---|
| T01 | 首次启动 | 在无实例状态启动程序 | 进入首次部署向导；无异常弹窗；主界面不显示旧实例信息 |  |
| T02 | 单实例稳定版部署 | 新建默认实例，不填实例名前缀，完成一次稳定版部署 | 部署成功；状态变为“运行中”；实例详情显示 `stable` |  |
| T03 | 稳定版镜像校验 | 打开默认实例部署目录中的 `docker-compose.yml` | Nekro Agent 镜像为 `kromiose/nekro-agent:latest` |  |
| T04 | 单实例预览版切换 | 在默认实例执行“切换至预览版”，选择“带备份” | 切换成功；实例详情显示 `preview`；恢复正式版入口可用 |  |
| T05 | 预览版备份文件校验 | 在 WSL 内检查备份归档文件 | 备份文件存在；路径与当前实例名匹配 |  |
| T06 | 预览版镜像校验 | 查看当前实例 `docker-compose.yml` | Nekro Agent 镜像为 `kromiose/nekro-agent:preview` |  |
| T07 | 恢复正式版 | 执行“恢复正式版” | 恢复成功；实例详情回到 `stable`；恢复入口按规则变化 |  |
| T08 | 恢复后镜像校验 | 再次查看 `docker-compose.yml` | 镜像恢复为 `kromiose/nekro-agent:latest` |  |
| T09 | 预览版下新增实例 | 保持实例 A 为 `preview`，再创建实例 B | 实例 B 默认仍为 `stable`，不会继承 A 的 `preview` |  |
| T10 | 第二实例镜像校验 | 查看实例 B 的 `docker-compose.yml` | 镜像为 `latest`，不是 `preview` |  |
| T11 | 混合频道并存 | 准备实例 A=`preview`、实例 B=`stable` | 两个实例详情页各自显示正确频道 |  |
| T12 | 混合频道冷启动 | 完全退出程序后重新启动 | 两个实例都能正常启动；频道不串用 |  |
| T13 | 默认实例为 Preview 时重启 | 将 A 设为默认实例，A=`preview`，B=`stable`，重启程序 | A 仍为 `preview`；B 仍为 `stable` |  |
| T14 | 默认实例为 Stable 时重启 | 将 B 设为默认实例，A=`preview`，B=`stable`，重启程序 | A 仍为 `preview`；B 仍为 `stable` |  |
| T15 | 活跃实例切换 | 在运行中从 A 切到 B，再切回 A | 设置页、文件页、浏览器地址、日志绑定都同步切换 |  |
| T16 | 路径页同步 | 切换实例后打开“存储与路径”页 | `DATA`/`CONF` 路径显示为当前实例的目录 |  |
| T17 | 默认实例切换 | 在设置页修改默认实例 | 默认实例选择成功；仅默认启动顺序变化，不改变各实例频道 |  |
| T18 | 无备份切预览 | 将某实例切到预览版时选择“不备份” | 该实例不可恢复正式版；不会影响其他实例的恢复能力 |  |
| T19 | 备份能力隔离 | 一个实例有备份，一个实例无备份，来回切换查看 | 每个实例单独显示自己的恢复能力，不串状态 |  |
| T20 | 新实例部署失败回滚 | 人为制造端口冲突或其他失败条件后新建实例 | 新实例回滚并移除；原活跃实例状态完整恢复 |  |
| T21 | 回滚后运行态校验 | 失败回滚后检查原实例 | 原实例的频道、端口、部署信息、浏览器地址正确 |  |
| T22 | 删除非活跃实例 | 删除一个非当前活跃实例 | 删除成功；当前实例状态不变 |  |
| T23 | 删除活跃实例 | 删除当前活跃实例，但仍有其他实例存在 | 自动切到剩余实例；频道、端口、路径同步正确 |  |
| T24 | 删除最后一个实例 | 删除最后一个剩余实例 | 回到首次运行状态；全局运行态清空 |  |
| T25 | 配置文件落盘 | 检查 `config.json` | 每个实例有独立 `release_channel`、`preview_backup_available`、`deploy_info` |  |
| T26 | 全局活动态一致性 | 切换当前实例后检查 `config.json` 顶层字段 | 顶层 `release_channel`、端口、`deploy_info` 与当前活跃实例一致 |  |

## 建议记录的检查点

- `config.json`：实际应用数据目录下的配置文件。
- 实例部署目录：每个实例各自的 `docker-compose.yml`、`.env`。
- WSL 内备份文件：默认实例和命名实例的备份归档路径。
- UI 页面：设置页、文件页、日志页、内置浏览器。

## 建议优先执行顺序

1. `T02` 到 `T10`
2. `T12` 到 `T19`
3. `T20` 到 `T26`
