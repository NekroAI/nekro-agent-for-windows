# Nekro Agent Windows 启动器人工检查流程

本文用于验证 Windows 启动器、WSL 运行环境、Docker Compose、多实例、内置浏览器、在线更新和迁移流程。自动检查通过不代表人工检查通过；源文件中的任务项保持未勾选，每轮测试应复制一份到发布记录中填写。

## 1. 执行规则

### 1.1 结果定义

- `PASS`：实际结果完全符合预期，证据已保存。
- `FAIL`：功能、状态、提示或数据完整性不符合预期，必须关联缺陷。
- `BLOCKED`：因环境或外部服务无法执行，必须写明阻塞原因，不能按通过处理。
- `N/A`：本次变更确实不涉及，并由复核人确认原因。

每项测试完成后，将 `[ ]` 改为 `[x]`，并在记录中填写 `PASS`、`FAIL`、`BLOCKED` 或 `N/A`。失败场景至少保留 UI 截图、`debug.log`、失败命令、返回码和命令输出。

### 1.2 执行频率

| 触发条件 | 必须执行 |
| --- | --- |
| 每次代码提交 | 自动门禁；与改动直接相关的 P0/P1/P2 用例 |
| 合并主分支或发布候选 | 全部 P0、P1；按风险选择 P2 |
| 修改 WSL、迁移、删除、备份或恢复 | P0、P1，以及 P2-A/P2-B/P2-C |
| 修改更新器、镜像下载或签名校验 | P0、P1-F，以及 P2-D |
| 修改窗口、托盘、WebView2 或页面状态 | P0、P1-D/P1-E，以及 P2-E |
| 正式发布 | 全部 P0、P1、适用的 P2、P3 发布验收 |

### 1.4 安全要求

- [ ] 使用专用测试机、虚拟机或可丢弃实例；破坏性测试前创建快照。
- [ ] 备份应用数据目录中的 `config.json`、测试实例数据和必要凭据，并确认备份可读。
- [ ] 记录测试前后的 `wsl -l -v`、容器、Compose、Docker volume 和端口状态。
- [ ] 默认实例和长期使用的数据不得用于删除、磁盘满、损坏归档或迁移回滚测试。
- [ ] 配置失败优先通过测试副本或调试注入模拟；若修改 ACL，测试结束后立即恢复并验证。
- [ ] 端口、目录和实例名必须唯一；破坏性测试使用一次性实例 `C`。
- [ ] 迁移、备份和恢复使用带已知 SHA-256 的测试文件，操作后重新计算并比较。
- [ ] 故障注入不得替换系统 PowerShell、修改证书存储或向正式下载流量注入伪造内容。

## 4. P0 冒烟检查

P0 用于快速确认程序可启动、可管理已有实例，并且主要状态不串用；实际耗时取决于镜像缓存和网络环境。

- [ ] **P0-01 启动与后端检查** — 正常启动 GUI，再运行 `uv run python main.py --backend-check`。预期主窗口无异常；控制台分别报告 WSL、发行版、Docker 和 Compose 状态，且检查过程不意外创建或注销发行版。
- [ ] **P0-02 调试日志与基础布局** — 使用 `uv run python main.py --debug` 启动，切换所有主页面并缩放窗口。预期 `debug.log` 写入应用数据目录；侧栏、版本、按钮和滚动区域无重叠或截断。
- [ ] **P0-03 已有实例冷启动** — 准备已停止和运行中的实例，完全退出并重新启动。预期状态栏、操作按钮、实例频道和实际容器状态一致，不把旧的 `is_running` 当成真实外部状态。
- [ ] **P0-04 服务启停** — 对当前实例依次启动、停止、再次启动。预期耗时操作不阻塞 UI；日志持续更新；按钮最终恢复；失败提示包含具体动作和输出。
- [ ] **P0-05 稳定版部署** — 部署一个 Lite stable 测试实例。预期 Compose 成功、镜像为 `kromiose/nekro-agent:latest`，端口、路径、频道、凭据和部署信息落盘，重启后仍可管理。
- [ ] **P0-06 NapCat 部署** — 部署一个 NapCat 测试实例。预期 Nekro 与 NapCat 服务、端口和 Token 正确，内置浏览器可分别访问，凭据不会写入其他实例。
- [ ] **P0-07 多实例切换** — 在 A/B 间来回切换。预期总览、设置卡、频道、端口、路径、日志源、浏览器地址、凭据和 NapCat 可见性全部绑定当前实例。
- [ ] **P0-08 更新基本链路** — 执行一次镜像检查和一次“启动器已是最新”检查。预期任务正常结束，状态和按钮恢复，可立即再次操作。
- [ ] **P0-09 忙状态互斥** — 在部署或更新中尝试切换实例、再次更新和退出。预期出现明确拦截，不启动重复任务；任务结束后互斥槽和按钮恢复。
- [ ] **P0-10 内置浏览器** — 打开、切换、刷新和关闭标签，访问 Nekro/NapCat。预期标题、地址、前进后退和页面状态正常，关闭最后一个标签时仍保留可用页面。

## 5. P1 核心发布回归

### P1-A 首次运行、WSL 与运行时

- [ ] **P1-A01 首次运行** — 在无实例状态启动。预期进入首次部署向导，不显示旧实例信息；取消后可重新进入。
- [ ] **P1-A02 环境检测** — 在 WSL、发行版、Docker 和 Compose 分别缺失或停止时检测。预期各项独立显示真实状态，失败项提供下一步操作，不把外部状态缓存为永久结果。
- [ ] **P1-A03 创建运行环境** — 完成 rootfs 下载、`wsl --import`、`wsl.conf`、配置保存和 Docker 安装。预期步骤和进度可见，失败时按钮恢复，成功后临时文件和安装标记按规则清理。
- [ ] **P1-A04 半安装续跑** — 分别在 guest marker/WSL 配置写入、启动器配置保存和 Docker 安装阶段制造失败，然后用同一流程重试；另覆盖首次发行版探测假阴性、import 返回“名称已存在”。预期只有能由本次 token/marker 证明归属的发行版才继续配置；无法证明归属时绝不自动 unregister，并保留恢复信息、提示人工核查。
- [ ] **P1-A05 既有发行版保护** — 准备无 guest marker 或 marker/token 不匹配的同名发行版。预期启动器拒绝自动复用或 `wsl --unregister`，并明确提示人工处理；原数据不变。
- [ ] **P1-A06 WSL 安装两阶段** — 在未启用 WSL 的快照环境执行安装，覆盖需要重启和重启后的第二阶段。预期不静默重启；重启后可重新检测并继续。
- [ ] **P1-A07 UAC 拒绝** — 在安装 WSL 的 UAC 中选择“否”。预期向导结束等待、按钮恢复、日志说明用户拒绝权限，界面不卡死。
- [ ] **P1-A08 Docker 初始停止** — Docker inactive 时启动或部署。预期按需启动；启动失败时不继续 Compose，错误包含 Docker 操作和输出。
- [ ] **P1-A09 外部状态消失** — 应用运行期间终止 WSL、Docker 或容器，再刷新或执行操作。预期重新探测后给出可操作错误，UI 不永久停留在“运行中”或忙状态。

### P1-B 部署、端口与配置持久化

- [ ] **P1-B01 Lite/NapCat 部署矩阵** — 分别部署默认/命名实例与 Lite/NapCat。预期 Compose 模板、服务、镜像、端口、凭据、部署目录和数据目录正确且互相隔离。
- [ ] **P1-B02 端口输入校验** — 输入文本、`0`、`65536`、同实例重复端口和另一实例已使用端口。预期保存或部署前阻止，并指出具体端口和冲突原因。
- [ ] **P1-B03 Windows 端口占用** — 用本地 HTTP 服务独占一个端口后部署。预期端口被判定为占用；检测不得因 `SO_REUSEADDR` 误判为空闲。停止占用进程后可重试成功。
- [ ] **P1-B04 运行中修改端口** — 运行中保存原端口，再改为新端口。预期原端口允许保存；新端口提示需重新部署后生效，当前运行页面仍指向实际旧端口，重新部署后才切换。
- [ ] **P1-B05 部署目录冲突** — 已存在默认目录时再次留空实例名，或预置目标目录。预期 Compose 启动前阻止，不合并目录，不产生第二条半成品实例记录。
- [ ] **P1-B06 部署失败回滚** — 分别在 pull、Compose up 和健康检查阶段制造失败。预期新实例按规则回滚，原活跃/默认实例及顶层兼容字段恢复，可直接重试。
- [ ] **P1-B07 新实例保存失败** — 在新实例配置首次落盘时模拟只读目录、磁盘满或保存函数返回失败。预期不启动 Compose；提示“实例配置保存失败”及原因；内存和磁盘不留下 active/default/实例半状态。
- [ ] **P1-B08 服务启动后保存失败** — 让 Compose 成功后，在保存凭据或 `deploy_info` 时失败。预期 UI 明确说明“服务已启动但配置未保存”，不显示完整部署成功或声称凭据已保存，并按真实容器状态刷新。
- [ ] **P1-B09 配置原子性** — 连续修改备注、默认实例、端口和高级功能后立即检查并重启。预期 `config.json` 始终为完整 JSON；实例字段和当前实例对应的顶层兼容字段一致。
- [ ] **P1-B10 配置保存失败的 UI** — 分别让默认实例、备注和普通设置保存失败。预期 UI 不假更新，不弹成功提示；重载页面后仍显示磁盘中的旧值，并包含可诊断错误。
- [ ] **P1-B11 损坏配置恢复** — 用非法或截断的测试 `config.json` 启动程序。预期原文件被保留为 `config.json.corrupt.<时间>`，程序使用默认配置继续启动并给出可诊断信息；不得覆盖或静默丢弃损坏文件。

### P1-C 多实例、频道、备份与删除

- [ ] **P1-C01 多实例混合频道** — 准备 A=`preview`、B=`stable`，轮换 active/default 后冷启动。预期各实例频道、镜像、备份能力和页面状态不串用；新实例默认仍为 stable。
- [ ] **P1-C02 带备份切预览版** — 对 A 选择带备份切换。预期备份路径按实例隔离，归档存在且可读；镜像为 `preview`；实例和顶层频道同步；恢复入口仅对 A 可用。
- [ ] **P1-C03 无备份切预览版** — 对 B 选择不备份。预期明确提示不能恢复；只把 B 的备份能力设为 false，不影响 A。
- [ ] **P1-C04 恢复 stable** — 从 A 的备份恢复。预期仅恢复白名单内的当前实例数据，镜像回到 `latest`，服务健康，频道与备份状态正确落盘。
- [ ] **P1-C05 频道保存失败** — 分别在 Compose 已切 preview、已恢复 stable 后模拟配置保存失败。预期任务报告失败并说明“实际运行状态已变化、配置未同步”，绝不误报成功；页面按实际服务重新探测。
- [ ] **P1-C06 更新阶段失败** — 在备份、pull、重建和健康检查阶段分别失败。预期服务实际状态被重新探测，按钮和状态栏恢复，不保留永久忙状态。
- [ ] **P1-C07 删除非活跃实例** — 删除一次性实例 C。预期确认框准确列出资源；Compose/部署目录按规则删除、用户数据按规则保留；A/B 的状态不变。
- [ ] **P1-C08 删除活跃实例** — 删除 active 实例且仍有其他实例。预期一次配置提交完成删除和 active 回退，随后切到剩余实例并重新探测状态；浏览器、日志和顶层兼容字段同步。
- [ ] **P1-C09 删除最后实例** — 删除最后一个实例。预期回到首次运行状态，实例、active/default 和兼容运行态清空；保留的数据明确告知用户。
- [ ] **P1-C10 删除后配置同步失败** — 只对一次性实例模拟“外部资源已删除、配置保存失败”。预期提示真实结果和修复建议，不显示“移除成功”；重启后的旧配置风险可识别，不自动对不存在资源执行危险操作。
- [ ] **P1-C11 active/default 分离** — A 为 active 时把 B 设为 default。预期当前界面不立即切到 B；完全退出再启动后优先展示 B，且日志、浏览器、凭据和顶层兼容字段同步到 B。

### P1-D 窗口、托盘、日志与设置

- [ ] **P1-D01 页面和实例联动** — 切换实例后逐页检查总览、设置、存储、日志和浏览器。预期路径、模式、频道、端口、凭据和日志立即绑定当前实例。
- [ ] **P1-D02 路径打开** — 从总览/存储页打开数据与部署目录。预期资源管理器打开当前实例对应路径；路径不存在时提示具体路径和处理建议。
- [ ] **P1-D03 日志切换** — 切换应用、Nekro、NapCat 和更新日志。预期内容刷新且不串实例；服务或日志文件不存在时页面保持可用。
- [ ] **P1-D04 关闭窗口三分支** — 存在实例时点关闭，分别在服务全停、混合运行和停止失败条件下测试取消、最小化到托盘、停止服务并退出。预期取消不改变状态；托盘可恢复；退出等待应停止的服务完成，并对停止失败给出明确处理。
- [ ] **P1-D05 托盘操作** — 测试托盘双击、“显示主窗口”和“退出”。预期窗口恢复一致；退出不留下启动器进程，停止策略与提示一致。
- [ ] **P1-D06 忙时关闭与切换** — 在部署、更新和配网时关闭窗口、退出或切换实例。预期明确拦截或安全取消，不产生重复线程；结束后按钮恢复。
- [ ] **P1-D07 开机自启** — 开启/关闭开机自启后重启程序并核对注册表。预期 UI、配置和系统项一致；写入失败时不显示成功。
- [ ] **P1-D08 高级功能开关** — 开关高级功能。预期预览版入口显示/隐藏正确；不会改变任何实例已有频道。

### P1-E WebView2 与凭据

- [ ] **P1-E01 标签与导航** — 在普通 UI 测试 Nekro/NapCat 目标切换、后退、前进和刷新；再用调试入口或测试 harness 创建多个标签，测试移动、切换、关闭和关闭最后标签。预期导航与标签状态稳定，最后仍有可用页面。
- [ ] **P1-E02 SPA 地址和标题同步** — 在能显示桥接回调结果的测试 SPA 中触发 `pushState`、`replaceState`、hash、`popstate` 和动态标题。预期地址栏与标题实时同步；SPA 显示通过 `window.qtwebview2.api` 收到的回调，且无脚本错误。
- [ ] **P1-E03 外部浏览器当前地址** — SPA 内导航后选择系统浏览器打开。预期打开当前真实 URL，不是初始 localhost 地址。
- [ ] **P1-E04 凭据填充** — Lite 登录页填充管理员凭据，NapCat 登录页填充 token。预期成功回调正确；无表单时给出明确提示；A/B 间不串凭据。
- [ ] **P1-E05 登录态和升级强刷** — 登录后重启启动器，再完成一次服务升级。预期浏览器 profile 保持登录态；升级后强制刷新获得新前端资源，地址与标签仍同步。
- [ ] **P1-E06 WebView2 初始化失败** — 仅在可回滚 VM/快照中使 Runtime 缺失或不可用后启动。预期显示可理解的安装/修复提示，主窗口其他功能可用且不崩溃；随后恢复快照或重新安装 WebView2 Runtime，并验证浏览器恢复正常。

### P1-F 镜像、启动器更新与 daemon

- [ ] **P1-F01 镜像列表与单项拉取** — 检查镜像状态并拉取一个缺失/可更新镜像。预期只显示相关镜像；本地/远程状态、进度、失败信息和完成后的按钮正确。
- [ ] **P1-F02 全部镜像更新** — 执行全部更新，并让其中一个拉取失败。预期逐项反馈，不因单项失败永久占用互斥槽。
- [ ] **P1-F03 启动器无更新** — 当前版本不低于 latest 时检查。预期明确提示已是最新，不在更新目录创建安装包或临时 exe。
- [ ] **P1-F04 daemon 通道** — 启动服务后检查桥接日志、监听端口、`.env` 和 `.na-tools/daemon.json`/`daemon.token`。预期 provider、端口和 token 权限正确，敏感 token 不进入普通日志。
- [ ] **P1-F05 daemon 端口冲突** — 启动第二个启动器进程，使其 daemon 尝试绑定相同端口。预期第二个进程明确提示端口/占用原因，现有 daemon 不受影响。
- [ ] **P1-F06 WebUI stable 更新** — 从 WebUI 勾选备份并更新。预期任务经历 validate/backup/pull/restart/verify，SSE 实时更新，健康检查成功后启动器状态同步。
- [ ] **P1-F07 SSE 续接与互斥** — 更新中刷新 WebUI，并交叉从启动器发起更新。预期 `after_seq` 续接不重复；启动器与 WebUI 互斥，任务完成后事件流收尾。
- [ ] **P1-F08 daemon 取消与重启恢复** — 取消任务并重启启动器。预期任务标记取消，状态栏恢复真实运行态；历史任务可查，未完成任务不永久悬挂，通道自动重建。
- [ ] **P1-F09 daemon 配置同步失败** — 外部更新成功后模拟配置保存失败。预期任务为 `failed/config_sync_failed`；服务健康时仍显示运行中；错误说明配置未同步且不误报成功。
- [ ] **P1-F10 WebUI 备份还原** — 创建备份、检查名称/时间/大小并还原。预期停止、白名单清理、解包、重启和健康检查顺序正确，当前实例以外的数据不变。

## 6. P2 扩展与故障注入

P2 使用快照和一次性数据执行。故障解除后必须再跑对应正常路径，确认可重试且无残留忙状态。

### P2-A 配置、磁盘与外部状态

- [ ] **P2-A01 配置目录只读** — 覆盖新部署、接管、频道切换、删除后的配置保存。预期任何流程都不假成功；内存与磁盘保持可解释状态；外部已完成动作被明确说明。
- [ ] **P2-A02 磁盘空间不足** — 覆盖 rootfs、迁移归档、备份和 Compose 文件写入。预期错误指出动作和目标路径，不产生空/半截 JSON，不把截断归档当作有效备份。
- [ ] **P2-A03 WSL 中途消失** — 在部署、更新和迁移时终止或注销测试发行版。预期失败提示包含 WSL 命令及输出，UI 解锁，不自动注销其他发行版。
- [ ] **P2-A04 Docker/Compose 中途消失** — 操作中停止 Docker 或删除测试容器。预期重新探测真实状态，失败可重试，不用陈旧缓存报告成功。

### P2-B 迁移与接管

- [ ] **P2-B01 扫描与托管发行版接管** — 扫描已有部署，并接管同一托管发行版内的实例。预期进度可结束；仅同步必要配置，不重复复制数据；保存失败不显示成功。
- [ ] **P2-B02 外部发行版迁移** — 迁移运行中的 Lite/NapCat 测试实例。预期仅停止实际运行的源服务，完成后 deploy/data/volume 校验值一致；迁移成功提交后源服务保持停止，避免源/目标双写。
- [ ] **P2-B03 扫描状态过期** — 扫描时让源实例为 stopped，迁移开始前从外部启动源服务。预期执行前实时探测并停止实际运行服务；若在提交前失败，只恢复迁移开始时实际运行的服务。
- [ ] **P2-B04 源停止失败** — 让源 Compose 停止失败。预期打包和目标写入均不开始，源保持原状态，错误含失败命令输出。
- [ ] **P2-B05 目标冲突预检** — 分别预置同名 deploy、data、Postgres volume 或 Qdrant volume。预期在停止源实例前阻断，任何目标不被覆盖或合并。
- [ ] **P2-B06 staging 安全解压** — 检查归档仅解压到独立 staging；使用包含绝对路径或 `..` 的测试归档。预期不直接向 `/` 释放，非法成员被拒绝，最终目录不受影响。
- [ ] **P2-B07 Windows TEMP 中转** — 使 `/mnt/wsl` 不可用。预期改用 Windows TEMP 完成中转，成功后清理，失败时保留足够诊断信息。
- [ ] **P2-B08 路径移动失败回滚** — 数据目录移动成功后让部署目录移动失败。预期已移动项逆序回到 staging，目标无半迁移内容，Docker 回到初始状态，可立即重试。
- [ ] **P2-B09 配置提交失败回滚** — 目录和 volume 移动后模拟配置保存失败。预期未提交迁移全部逆序回滚，源实例恢复，目标不留下可被误识别的实例。
- [ ] **P2-B10 回滚失败保全** — 占用 staging 回滚目标或让 `mv` 失败。预期不覆盖 staging、不清理恢复材料，Docker 保持安全状态，日志列出源、目标和失败命令。
- [ ] **P2-B11 迁移提交后 Docker 恢复失败** — 目录、volume 和配置同步成功后让目标 Docker 启动失败。预期迁移视为已提交，不回滚或删除目标数据，不重新启动源实例造成双写；提示用户手动恢复目标 Docker，并准确说明 staging 和服务状态。

### P2-C Docker volume 与归档恢复

- [ ] **P2-C01 volume 间外部启动 Docker** — 第一个 volume 移动后启动 Docker。预期第二个 volume 前重新检查、停止并验证 Docker inactive；验证失败时不继续移动。
- [ ] **P2-C02 回滚间外部启动 Docker** — 第一个 volume 回滚后再次启动 Docker。预期下一项回滚前再次复核，而不是复用一次性的状态结果。
- [ ] **P2-C03 Docker stop 失败** — 让 `systemctl stop docker` 失败。预期立即中止后续 volume 变更；尚未涉及 volume 的目录尽可能安全回滚；无法继续回滚时保留 staging 和现有数据位置，明确列出未回滚路径，且不得误报完成。
- [ ] **P2-C04 初始 inactive 保持** — Docker 初始 inactive，迁移期间被外部启动。预期为 volume 安全停止后最终仍为 inactive，不被错误恢复为 active。
- [ ] **P2-C05 损坏备份/迁移归档** — 使用截断或篡改的测试归档。预期校验或解压失败，不覆盖现有实例；源服务和 UI 状态重新探测。
- [ ] **P2-C06 旧混合归档白名单** — 使用旧版生成且包含其他实例路径的归档恢复命名实例。预期仅处理当前实例允许成员，默认实例及其他 volume 校验值不变。

### P2-D 启动器更新包安全与取消

- [ ] **P2-D01 API/下载网络失败** — 仅对测试进程设置无效代理或阻断测试地址。预期提示包含 API/下载阶段、来源、HTTP 或网络详情；UI 不冻结且可重试。
- [ ] **P2-D02 Release 资产选择** — 测试 Release 同时包含 `.exe` 和 `.exe.sha256` sidecar。预期只选择真正安装程序，不把 sidecar 当作可执行资产。
- [ ] **P2-D03 大小和内容校验** — 在临时 harness 中返回 Content-Length/Release size 不符、截断字节或 HTML 内容。预期拒绝，删除临时文件，不启动安装器。
- [ ] **P2-D04 SHA-256 与镜像信任** — 测试正确/错误 digest、镜像成功/失败和 GitHub 回退。预期错误 digest 被拒绝；有官方 digest 才允许第三方镜像；无 digest 时不得尝试镜像。
- [ ] **P2-D05 PE 与 Authenticode** — 验证真实签名包、篡改签名包和测试无签名包。预期 MZ/PE 无效必拒绝；Authenticode Invalid 必拒绝；无签名官方包仅按既定 size/来源策略处理。
- [ ] **P2-D06 下载取消和立即退出** — 用本地延迟 HTTP 服务模拟慢下载，按 Esc 后立即关闭程序。预期无崩溃、无 `QThread destroyed while running`，重新打开可再次下载，更新目录无不可解释残留。

### P2-E UI、WebView 和并发状态机

- [ ] **P2-E01 pull 静默超时** — 拉取时断网并保持无输出。预期超时后提示“镜像拉取超时”，互斥槽释放，按钮可重试。
- [ ] **P2-E02 向导遇到主窗口忙** — 主窗口有更新任务时从向导继续部署。预期提示占用并回到可重试步骤，不永久显示“部署中”。
- [ ] **P2-E03 升级中切换实例** — 更新中尝试切换 active 实例。预期被阻止，更新仍作用于原实例；完成后结果判定正确。
- [ ] **P2-E04 删除目录缺失的实例** — 手工移走测试部署目录后执行卸载/删除。预期跳过不存在的目录并继续安全清理，提示实际完成和保留项。
- [ ] **P2-E05 凭据等待中关闭** — WebUI 尚未就绪时关闭等待或凭据弹窗。预期后台线程安全结束，按钮恢复，不影响后续重试。

### P2-F 故障注入实施要求

| 场景 | 推荐方法 | 恢复与核验 |
| --- | --- | --- |
| 配置保存失败 | 使用测试配置副本、调试开关或定点 mock 让保存接口返回失败；避免修改长期使用的真实目录 | 解除注入，确认旧 JSON 完整可读，再执行一次正常保存 |
| volume 阶段暂停 | 在测试 harness/调试器中暂停于相邻 volume 操作之间，再从外部改变 Docker 状态 | 恢复执行前记录 `systemctl is-active docker`；结束后核对全部 volume 和 staging |
| Docker stop/start 失败 | 对测试发行版使用受控 service wrapper 或定点 mock；不要破坏系统级 Docker 安装 | 移除 wrapper/mock，手工验证 Docker 可启停，再跑正常迁移 |
| 恶意或损坏归档 | 只替换一次性实例的测试归档，并记录替换时机、路径和原 SHA-256 | 恢复原归档或快照，比较源/目标数据 SHA-256 |
| 更新包截断/伪造 | 使用本地 HTTP 服务和进程内 monkeypatch，禁止替换系统证书或污染正式下载源 | 关闭本地服务，清理临时更新目录，再执行一次官方无更新/正常下载检查 |

每个故障注入记录必须写明注入点、触发方式、解除步骤和解除后的正常路径结果；无法稳定重复的故障不得仅凭一次现象判定通过。
